## 1. Общее описание и технологии

### Описание

«Голодные змейки 🐍» — браузерная 2D-игра на HTML5 Canvas. Игрок управляет змейкой, которая перемещается по клеточному полю, поедает еду (базово — яблоки), избегает стен и других змеек. Игра поддерживает от 0 до 2 игроков-людей и от 0 до 4 ботов одновременно. Игра разделена на уровни с нарастающей сложностью.

### Технологии

| Параметр | Значение |
|---|---|
| Язык | TypeScript (strict mode) |
| Сборка | webpack (dev server + production build) |
| Рендер | HTML5 Canvas 2D API |
| Тесты | Jest + ts-jest (среда jsdom) |
| Вёрстка | Адаптивная, игровое поле по центру экрана |
| Стили | Только CSS-классы и CSS custom properties (переменные). **Inline-стили запрещены** |
| Интерфейс | Полностью на **русском** языке |
| Хранение данных | `localStorage` (рекорды, имена игроков, настройки dev-режима) |
| Округление | `Math.floor()` для всех вычислений, дающих дробное число |

### Принцип модульности

Проект разделён на два независимых слоя:

- **Engine** — чистая логика: состояние игры, движение, коллизии, спавн, формулы. Не зависит от DOM и Canvas. Покрывается unit-тестами.
- **Renderer** — отрисовка текущего состояния на Canvas. Не содержит игровой логики.

Все сущности (змейки, еда, стены) хранятся в отдельных массивах внутри `GameState`. Двумерный массив `board[][]` пересобирается из сущностей каждый тик для отладки и AI.

## 2. Режимы игры

### Комбинации участников

Поддерживаются любые комбинации, удовлетворяющие ограничениям:

| Параметр | Допустимые значения |
|---|---|
| Игроки-люди | **0, 1 или 2** (не более 2) |
| Боты (AI) | **0, 1, 2, 3 или 4** |
| Суммарно змеек | От 1 до 6 |

#### Режим наблюдателя (0 игроков)

Если игроков-людей **0**, боты играют между собой. Пользователь только наблюдает. Управление клавиатурой не привязывается ни к одной змейке. Пауза по пробелу всё равно доступна.

#### Одиночный режим

Одиночный режим имеет 2 подрежима:

1. **Классика (1 игрок, 0 ботов)**:
   - цель — пройти уровни, набирая кумулятивный порог очков;
   - общее количество уровней: **10**;
   - после завершения 10-го уровня игра заканчивается.
2. **Выживание (1 игрок, 0 ботов)**:
   - общее количество уровней: **100**;
   - переход на следующий уровень происходит при достижении целевого счёта текущего уровня;
   - переход выполняется автоматически, **без паузы и без модалки конца уровня**;
   - после успешного завершения уровня 100 игра заканчивается победой.

#### Многопользовательский / смешанный режим

Любая комбинация с ≥2 змейками. Уровень заканчивается по выживаемости или таймеру. После уровня 10 игра завершается и определяется итоговый победитель по победам и очкам.

### Сложность

| Параметр | Значение |
|---|---|
| Диапазон | **1–10** (целое число) |
| По умолчанию | **1** |
| Выбирается | В меню перед началом игры |

Сложность влияет на:
- Длину генерируемых стен (больше → длиннее)
- Количество начальной еды (больше → меньше еды)

## 3. Управление

### Управление змейками

Игра поддерживает одновременное управление двумя игроками на одной клавиатуре:

| Игрок | Вверх | Влево | Вниз | Вправо |
|-------|-------|-------|------|--------|
| Игрок 1 | `W` | `A` | `S` | `D` |
| Игрок 2 | `↑` (ArrowUp) | `←` (ArrowLeft) | `↓` (ArrowDown) | `→` (ArrowRight) |

Каждый набор клавиш привязывается к соответствующей змейке по порядку: первый человек в списке — WASD, второй — стрелки.

Дополнительное правило:
- В режиме **1 игрок** управление первой змейкой принимается **одновременно** и от WASD, и от стрелок.
- В режиме **2 игрока** схема остаётся классической: игрок 1 — WASD, игрок 2 — стрелки.

#### Правила обработки ввода

1. **Запрет разворота на 180°**: если змейка движется вправо, нажатие «влево» игнорируется. Аналогично для всех направлений. Змейка сохраняет прежнее направление.
2. **Одно направление за тик**: если игрок нажал несколько клавиш между тиками, применяется **последнее** нажатие (которое не является разворотом на 180°).
3. **Направление применяется в начале тика**, до расчёта движения.

### Touch-управление

При обнаружении touch-устройства показываются onscreen-кнопки управления.

#### Режим 1 игрок

- В левом нижнем углу игрового блока: кнопки `←` и `→`.
- В правом нижнем углу игрового блока: кнопки `↑` и `↓`.
- Кнопки работают как дополнительный источник направления для игрока 1 и подчиняются тем же правилам буфера ввода (последнее валидное направление за тик).

#### Режим 2 игрока

Touch-раскладка для двух игроков (кнопки для обеих сторон экрана без поворота HUD) реализуется отдельным этапом и уточняется в разделе 18.

### Пауза

| Клавиша | Действие |
|---------|----------|
| **Space** (Пробел) | Контекстное действие — зависит от текущего состояния |
| **Escape** | Контекстное действие выхода/отмены (см. ниже) |
| **Enter** | Подтверждение в модалках выбора `Да / Нет` |

Поведение клавиши «Пробел»:

| Текущее состояние | Что происходит при нажатии Space |
|-------------------|----------------------------------|
| Игра идёт (нет модалки) | Ставит игру на **паузу**: game loop останавливается, появляется модальное окно «ПАУЗА» |
| Модалка «ПАУЗА» | Снимает паузу: модалка закрывается, game loop продолжает работу |
| Модалка «Конец уровня» | Закрывает модалку и переходит к следующему уровню (аналогично кнопке «Продолжить») |
| Модалка «Игра окончена» | Закрывает модалку и переходит на экран результатов |

Примечание: в режиме выживания модалка «Конец уровня» не используется, поэтому переход по Space для этого состояния не применяется.

Поведение `Escape`:
- В состоянии **Playing**: игра ставится на паузу и открывается модалка подтверждения выхода (`Да/Нет`).
- В состоянии **Paused**: мгновенный выход в главное меню.
- На экране **Results**: возврат в главное меню.
- В модалке подтверждения: `Enter` = подтвердить выход (`Да`), `Escape` = отменить (`Нет`) и вернуться в игру.

Во время паузы:
- Game loop полностью остановлен (тики не идут)
- Таймер уровня не убывает
- Canvas замораживается на последнем кадре
- Поверх поля появляется модальное окно с затемнённым overlay

В **dev-режиме** модалка паузы покрывает **только игровую область** (canvas, HUD-панели), но **не панель настроек** справа — она остаётся кликабельной.
## 4. Игровое поле

### Размер поля

Размер поля зависит от текущего уровня:

```
ширина = baseWidth  + (level − 1) × levelSizeIncrement
высота = baseHeight + (level − 1) × levelSizeIncrement
```

Значения по умолчанию (из `gameDefaults.json`):

| Параметр | Значение |
|---|---|
| `baseWidth` | 40 |
| `baseHeight` | 40 |
| `levelSizeIncrement` | 5 |

Примеры: уровень 1 — 40×40, уровень 2 — 45×45, уровень 3 — 50×50.

### Внутреннее представление (engine)

Поле представлено двумерным массивом `CellContent[][]` (строковые маркеры):

| Символ | Значение |
|--------|----------|
| `' '` (пробел) | Пустая клетка |
| `'&xN'` | Еда, где `N` — очки за поедание текущей фазы (например, `&x2` для взрослого яблока) |
| `'*'` | Стена / преграда |
| `'1'..'6'` | Сегмент тела змейки по `snake.id + 1` (`1` — первая змейка, `2` — вторая, `3..6` — боты) |

Массив `board[][]` **пересобирается из сущностей** каждый тик (не является мастер-данными). Мастер-данные — отдельные массивы `state.walls`, `state.foodItems`, `state.snakes`.

### Границы поля

Границы поля являются непроходимыми: выход за пределы координат `[0, width)` × `[0, height)` считается столкновением со стеной и вызывает смерть змейки. Отдельных стен по периметру нет — граница неявная.

### Координатная система

- Ось **X** — горизонтальная, растёт слева направо (0 = левый край).
- Ось **Y** — вертикальная, растёт сверху вниз (0 = верхний край).
- Клетка `(x, y)` рисуется как прямоугольник `[x × cellSize, y × cellSize, cellSize, cellSize]`.

## 5. Змейки

### Структура данных

Каждая змейка хранится как объект `Snake`:

```typescript
interface Snake {
  id: number;               // уникальный идентификатор (0, 1, 2…)
  name: string;             // имя (из меню или «Бот N»)
  segments: Position[];     // массив координат, segments[0] = голова
  direction: Direction;     // текущее направление ('up' | 'down' | 'left' | 'right')
  alive: boolean;           // жива ли
  score: number;            // очки (суммарная ценность съеденной еды)
  levelsWon: number;        // количество выигранных уровней
  ticksWithoutFood: number; // тиков без еды (для механики голода)
  isBot: boolean;           // управляется ли AI
  deathReason?: string;     // причина смерти (если мертва)
}
```

### Начальные параметры

| Параметр | Значение по умолчанию |
|---|---|
| Начальная длина | **5** сегментов (`initialSnakeLength`) |
| Начальное направление | Зависит от стартовой позиции |
| `ticksWithoutFood` | 0 |
| `score` | 0 |
| `levelsWon` | Переносится между уровнями |

### Стартовые позиции

Змейки размещаются на заранее рассчитанных позициях, распределённых по полю. Стартовые позиции зависят от размера поля и количества змеек. Между стартовой позицией и краем поля оставляется отступ не менее `initialSnakeLength + 2` клеток.

Тело змейки при старте вытянуто в направлении, **противоположном** начальному направлению движения. Пример: если змейка стартует в позиции `(7, 20)` с направлением `right`, её голова в `(7, 20)`, следующий сегмент в `(6, 20)`, потом `(5, 20)` и т.д.

### Движение

1. Каждый тик змейка перемещается ровно на **1 клетку** в текущем направлении.
2. Перемещение реализовано как:
   - Вычисление координаты новой головы (`getNextHeadPosition`).
   - Добавление новой головы в начало массива `segments`.
   - Удаление последнего сегмента (хвоста) — если змейка НЕ ест еду.
   - Если змейка ест еду — хвост НЕ удаляется, змейка растёт на `food.growthValue`.

### Смерть змейки

Змейка погибает (поле `alive = false`) в следующих случаях:

| Причина | Когда проверяется | `deathReason` |
|---------|-------------------|---------------|
| **Столкновение со стеной** | Новая голова попала на стену или за границу поля | `'Врезалась в стену'` |
| **Столкновение с другой змейкой** | Новая голова попала на сегмент другой живой змейки | `'Столкнулась с другой змейкой'` |
| **Самоукус** | После перемещения голова совпадает с любым другим сегментом той же змейки | `'Съела саму себя'` |
| **Голод** | После потери сегмента длина стала < `minSnakeLength` (по умолчанию 2) | `'Умерла с голоду'` |

#### Порядок проверок в тике:

1. Вычисляется новая позиция головы.
2. Проверяется столкновение со стеной → если да, змейка погибает, **пропуск оставшихся шагов**.
3. Проверяется столкновение с другими змейками → если да, змейка погибает.
4. Выполняется перемещение (голова добавляется, хвост удаляется или нет).
5. Проверяется самоукус → если да, змейка погибает.
6. Если новая голова совпадает с позицией еды → еда съедается.

### Поедание еды

Когда голова змейки оказывается в клетке, где находится еда:

1. Еда **удаляется** из массива `state.foodItems`.
2. Змейка растёт на `food.growthValue` сегментов.
3. Счёт змейки увеличивается на `food.scoreValue`.
4. Счётчик голода **сбрасывается** (`snake.ticksWithoutFood = 0`) для **любой** еды.

### Голод

Механика голода стимулирует змейку активно искать пищу:

| Параметр | Значение по умолчанию |
|---|---|
| `hungerThreshold` | **15** тиков |
| `minSnakeLength` | **2** сегмента |

**Алгоритм (каждый тик, после движения):**

1. `ticksWithoutFood++`.
2. Если `ticksWithoutFood >= hungerThreshold`:
   - Удаляется последний сегмент (хвост) — змейка укорачивается на 1.
   - `ticksWithoutFood` сбрасывается в **0**.
   - Если `segments.length < minSnakeLength` → змейка **погибает** с причиной `'Умерла с голоду'`.

Таким образом, если змейка не ест, она теряет по 1 сегменту каждые `hungerThreshold` тиков, пока не останется менее 2 сегментов и не умрёт.

### Мёртвые змейки

После смерти змейка **не исчезает** с поля:

- Её тело остаётся на месте до конца уровня.
- Визуально мёртвая змейка отображается **серым цветом** (обесцвечивание).
- Коллизии с мёртвыми змейками **не проверяются** (другие змейки могут проходить сквозь мёртвое тело).

### Визуальное отображение змейки

#### Тело (живая змейка)

- Каждый сегмент занимает одну клетку (`cellSize × cellSize`).
- Заливка: **базовый цвет** змейки (из палитры `snakeColors`).
- Обводка каждого сегмента: тонкая линия (`lineWidth: 1`), цвет на **35% темнее** базового.

#### Голова (живая змейка)

- Заливка: базовый цвет, затемнённый на **25%** (темнее тела).
- Обводка: **белый** (`colorHeadStroke`, по умолчанию `#FFFFFF`), `lineWidth: 1`.
- Два **чёрных глазка**: маленькие круги (`radius = max(1, cellSize × 0.12)`), расположенные со стороны направления движения. Отступ от края клетки: `cellSize × 0.25`.
  - Направление `up` → глаза у верхнего края (левый и правый).
  - Направление `down` → глаза у нижнего края.
  - Направление `left` → глаза у левого края (верхний и нижний).
  - Направление `right` → глаза у правого края.

#### Мёртвая змейка

- Все сегменты (включая голову) закрашиваются **серым** (обесцвеченный базовый цвет: `grey = round(r×0.3 + g×0.3 + b×0.3)`).
- Обводка каждого сегмента: цвет на **35% темнее** серого.
- Глазки **не рисуются**.

#### Палитра змеек

Цвета назначаются по порядку (по `snake.id`):

| # | Цвет по умолчанию |
|---|---|
| 0 | `#00FF00` (зелёный) |
| 1 | `#00CCFF` (голубой) |
| 2 | `#FFFF00` (жёлтый) |
| 3 | `#FF00FF` (пурпурный) |
| 4 | `#FF8800` (оранжевый) |
| 5 | `#88FF88` (светло-зелёный) |

Если змеек больше чем цветов — палитра зацикливается.
## 6. Еда (Food)

### 6.1 Доменная модель еды

Еда реализуется через базовый класс `Food` и наследников (минимум `AppleFood`, `RabbitFood`).

```typescript
interface Food {
  id: string;
  kind: 'apple' | 'rabbit' | string;
  pos: Position;
  age: number;               // абсолютный возраст в тиках
  clockNum: number;          // тиков с рождения/последнего размножения
  reproductionCount: number; // сколько раз размножалась сущность
  phase: 'young' | 'adult' | 'old';
  scoreValue: number;        // сколько очков даёт при поедании
  growthValue: number;       // на сколько сегментов растёт змейка
}
```

Общие правила:
- возраст (`age`) и логика фаз — в базовом `Food`;
- размножение — общее поведение базового `Food`, с возможностью уточнения в наследниках;
- сброс голода змейки при поедании работает для **любой** еды.

Требование совместимости доменной логики:
- всё поведение, которое ранее было определено для кроликов (жизненный цикл, размножение, ограничения соседства, правила размещения и удаления по возрасту), считается **базовым поведением `Food`**;
- дочерние классы (`AppleFood`, `RabbitFood` и другие) могут расширять или переопределять это поведение.

### 6.2 Активные типы еды по режимам и уровням

- **Классика (1P) и мультиплеер (2+ змейки)**: уровни `1..10`, еда по умолчанию — `AppleFood`.
- **Выживание (1P Survival)**: уровни `1..100`, еда по умолчанию — `AppleFood`.
- `RabbitFood` остаётся поддерживаемым типом для расширений/экспериментов, но в базовом режиме не обязателен к включению в первые уровни.

### 6.3 Начальный спавн еды

При инициализации уровня генерируется начальное количество еды:

```
foodCount = Math.floor(foodCountPerSnakeCoeff × snakeCount + foodCountBase − difficultyLevel)
```

Значения по умолчанию: `foodCountPerSnakeCoeff = 1.5`, `foodCountBase = 10`.

Состав стартовой еды:
- количество **взрослой** еды (`phase = adult`) равно количеству змеек;
- остальная стартовая еда — `young`.

Правила размещения:
1. Не на стене.
2. Не на живой змейке.
3. Расстояние Чебышёва до любой другой еды > 1.
4. Внутри границ поля.

Если за `count × 100` попыток не удалось разместить всю еду — размещается столько, сколько получилось.

### 6.4 Жизненный цикл еды (общий)

У каждой еды есть абсолютный возраст `age`, который увеличивается на 1 каждый тик.

| Фаза | Диапазон `age` (по умолчанию) |
|------|-------------------------------|
| `young` | `0 ≤ age < 50` |
| `adult` | `50 ≤ age < 100` |
| `old` | `100 ≤ age < 150` |

На тике, когда `age >= maxAge` (по умолчанию `150`), еда удаляется с поля.

Параметры жизненного цикла:
- `foodYoungAge = 50`
- `foodAdultAge = 100`
- `foodMaxAge = 150`

### 6.5 Значения еды по умолчанию (`AppleFood`)

Для яблок используются 3 стадии и их ценность:

| Стадия | Фаза | Цвет | Очки | Рост |
|--------|------|------|------|------|
| Маленькое | `young` | Зелёный | `+1` | `+1` |
| Взрослое | `adult` | Красный | `+2` | `+2` |
| Старое | `old` | Коричневый | `+1` | `+1` |

### 6.6 Размножение еды (общее правило)

Размножение проверяется каждый тик для каждой еды.
Условия базового алгоритма:
1. Фаза `adult`.
2. `clockNum >= reproductionMinCooldown` (по умолчанию `5`).
3. `reproductionCount < maxReproductions` (по умолчанию `5`).
4. Если соседей в радиусе `neighborReproductionRadius` (Чебышёв) не меньше `maxReproductionNeighbors` (по умолчанию `4`) — размножение блокируется.

Вероятность:

```
baseProbability = reproductionProbabilityBase × clockNum
penalty = neighborReproductionPenalty × nearbyCount
probability = baseProbability × (1 − penalty)
```

При успешной проверке создаётся потомок того же типа на расстоянии 1–2 по Чебышёву при соблюдении стандартной валидации клетки.

### 6.7 Автодобавление взрослой еды

Если на поле еды меньше, чем живых змеек, выполняется автодобавление:
1. Добавляется **одна** еда в фазе `adult`.
2. Позиция выбирается как максимально удалённая от голов живых змеек:
   - оптимизируется минимум из дистанций Чебышёва до каждой головы (maximin);
   - при равенстве выбирается позиция с наименьшей разницей дистанций до голов (наиболее равноудалённая);
   - если равенство сохраняется — используется детерминированный tie-break (например, лексикографически по `y`, затем `x`).
3. Ограничение частоты: не чаще 1 раза за `hungerThreshold` тиков.

## 7. Генерация стен

### Параметры генерации

Количество кластеров и длина стен определяются формулами:

```
clusterCount = Math.floor(wallClusterCoeff × level + wallClusterBase)
wallLength   = Math.floor(wallLengthCoeff × difficultyLevel + wallLengthBase)
```

Значения по умолчанию: `wallClusterCoeff = 1.2`, `wallClusterBase = 2`, `wallLengthCoeff = 1.2`, `wallLengthBase = 3`.

Пример: уровень 3, сложность 5 → кластеров = `Math.floor(1.2 × 3 + 2) = 5`, длина = `Math.floor(1.2 × 5 + 3) = 9`.

### Алгоритм генерации (Random Walk + ветвления)

Для каждого кластера:

1. Выбирается **случайная стартовая точка** внутри поля (минимум 2 клетки от края).
2. От стартовой точки выполняется **случайное блуждание** (random walk): на каждом шаге случайно выбирается одно из 4 направлений (вверх/вниз/влево/вправо).
3. С вероятностью **30%** на каждом шаге создаётся **ответвление** — дополнительная стена в случайном направлении от текущей позиции.
4. Каждый новый сегмент проверяется:
   - Он не за пределами поля (не менее 1 клетки от границы).
   - Он не в **зоне безопасности** стартовых позиций змеек.
5. Если нет допустимых направлений — кластер обрывается.

### Зона безопасности змеек

Стены не могут генерироваться рядом со стартовыми позициями змеек.

**Радиус безопасности**: `Math.ceil(1.5 × initialSnakeLength)` клеток (по расстоянию Чебышёва).

При `initialSnakeLength = 5` → радиус = **8 клеток**. Стены не могут быть ближе 8 клеток (по Чебышёву) к любому сегменту стартового тела любой змейки.

Стартовые позиции змеек вычисляются **до** генерации стен. Зоны безопасности строятся по **всем** клеткам, которые будут заняты телами змеек при старте.

### Валидация стен (BFS-связность)

После генерации всех кластеров проверяется связность свободного пространства:

1. Находится **любая** свободная клетка на поле.
2. Запускается **BFS** (поиск в ширину) по свободным клеткам (4 направления: вверх/вниз/влево/вправо).
3. Подсчитывается количество достижимых клеток.
4. Если `visited.size === totalFree` (где `totalFree = width × height − wallCount`) → стены **валидны**.
5. Если нет → стены **невалидны** (есть изолированные зоны), нужна перегенерация.

**Правило**: стены не должны создавать изолированных комнат. Все свободные клетки поля должны быть связаны (достижимы друг из друга).

### Перегенерация

- Если валидация не прошла → генерируется новый набор стен.
- Максимум **50 попыток**.
- Если за 50 попыток не удалось сгенерировать валидные стены → поле создаётся **без стен** (fallback).

### Per-level overrides (dev mode)

В dev-режиме для конкретного уровня можно задать фиксированные значения `wallClusters` и `wallLength`, которые будут использоваться вместо формул.

### Визуальное отображение стен

- Каждая стена занимает одну клетку.
- Заливка: `colorWall` (по умолчанию `#CCCCCC` — светло-серый).
- Без обводки.
## 8. Победа, уровни, счёт

### Общая структура

Игра состоит из **уровней** (level). На каждом уровне:
1. Генерируется новое поле (стены, еда, стартовые позиции).
2. Очки (`score`) и победы (`levelsWon`) **переносятся** между уровнями.
3. Тик уровня (`tickCount`) и таймер (`levelTimeLeft`) — **обнуляются**.

### Одиночная игра (1 змейка)

#### Цель уровня

Формула цели для отдельного уровня (в очках):

```
targetScore(level) = Math.floor(targetScoreCoeff × level + targetScoreBase)
```

По умолчанию: `targetScoreCoeff = 5`, `targetScoreBase = 20`.

**Кумулятивная цель** — сумма целей всех пройденных уровней:

```
cumulativeTarget(level) = Σ targetScore(l) для l = 1..level
```

Примеры (при дефолтных коэффициентах):
- Уровень 1: цель = 25, кумулятивная = 25
- Уровень 2: цель = 30, кумулятивная = 55
- Уровень 3: цель = 35, кумулятивная = 90

#### Условия завершения уровня (классика, 1 игрок)

| Условие | Результат |
|---------|-----------|
| `snake.score >= cumulativeTarget(level)` | **Уровень пройден** — переход к следующему |
| `snake.alive === false` | **Игра окончена** (game over) |

При успешном прохождении:
1. Игра **останавливается** (game loop стоит).
2. Появляется модальное окно «Уровень N завершён» с промежуточными результатами.
3. Игрок нажимает «Продолжить» (или Space) → инициализируется следующий уровень.
4. После завершения уровня 10 игра считается завершённой.

#### Условия завершения уровня (выживание, 1 игрок)

Используются те же базовые правила жизни/смерти змейки и та же формула целевого счёта уровня (`targetScore` / `cumulativeTarget`).

Правила прогрессии:
1. При достижении цели текущего уровня уровень повышается немедленно.
2. Переход выполняется без остановки игры: без модалки и без паузы между уровнями.
3. После успешного завершения уровня 100 игра завершается победой игрока.

### Многопользовательская игра (≥2 змеек)

#### Условия завершения уровня

| Условие | Результат |
|---------|-----------|
| Осталась **1 живая** змейка | Она — **победитель** уровня, получает `levelsWon++` |
| **Все погибли** | **Нет победителя** (0 побед) |
| Таймер истёк (**3 минуты**), ≥2 живых | **Ничья** (никто не получает побед) |

#### Таймер уровня

- При инициализации уровня: `levelTimeLeft = levelTimeLimit` (по умолчанию **180** секунд = 3 минуты).
- Каждую секунду: `levelTimeLeft--`.
- При `levelTimeLeft <= 0` → уровень заканчивается.
- Во время паузы таймер **не идёт**.

### Определение итогового победителя

По окончании всех уровней (или при game over) определяется итоговый победитель по приоритету:

1. **Больше побед** (`levelsWon`) → победитель.
2. Если побед **поровну** → у кого **больше очков** (`score`) → победитель.
3. Если и побед, и очков поровну → **ничья** (нет победителя).

### Переход между уровнями

При переходе на следующий уровень:
- `level++`
- `score`, `levelsWon` — **сохраняются** от предыдущего уровня.
- `tickCount = 0`, `levelTimeLeft = levelTimeLimit`.
- Генерируются новые стены, еда, стартовые позиции.
- Все змейки **оживают** (`alive = true`, `deathReason` очищается).
- Длина змеек сбрасывается до `initialSnakeLength`.
- `ticksWithoutFood = 0`.

## 9. Боты (AI)

### Общая концепция

Боты — автономные змейки, управляемые эвристическим алгоритмом. Каждый тик бот анализирует видимую область вокруг головы и выбирает одно из трёх действий: повернуть влево, повернуть вправо или продолжить прямо.

### Интерфейс решения

```typescript
type BotDecision = 'left' | 'right' | 'front';

interface BotInput {
  vision: number[][];        // матрица visionSize × visionSize
  snakeLength: number;       // текущая длина змейки
  ticksWithoutFood: number;  // тиков без еды
}

function botDecide(input: BotInput): BotDecision;
```

Решение `BotDecision` — поворот **относительно текущего направления** змейки:
- `'front'` — продолжить в текущем направлении.
- `'left'` — повернуть налево (например, если змейка шла вправо → пойдёт вверх).
- `'right'` — повернуть направо.

Затем решение конвертируется в абсолютный `Direction` (`up/down/left/right`).

### Матрица vision

#### Размер

Квадратная матрица `visionSize × visionSize` (по умолчанию **20×20**).

#### Ориентация

Матрица **повёрнута** относительно текущего направления змейки так, что:
- **Направление змейки** всегда смотрит «вверх» (к row 0).
- Голова змейки находится в **центре** матрицы (`visionSize/2, visionSize/2`).

Утилита `rotateToWorld` переводит координаты матрицы обратно в мировые координаты.

#### Значения ячеек

Ячейки матрицы содержат **суммированные сигналы** от всех ближайших объектов:

**Сигналы препятствий** (стены, границы поля, тела змеек) — **отрицательные**:

| Расстояние (Чебышёв) | Сигнал |
|---|---|
| 1 клетка | `obstacleSignalClose` (по умолчанию **−100**) |
| 2 клетки | −80 |
| 3 клетки | −60 |
| d клеток | `obstacleSignalClose + obstacleSignalDecay × (d − 1)` |

Затухание: `obstacleSignalDecay = 20` за каждую клетку расстояния. Сигналы от нескольких препятствий **суммируются**.

**Сигналы еды** — **положительные**:

| Расстояние (Чебышёв) | Сигнал |
|---|---|
| 1 клетка | `foodSignalClose` (по умолчанию **+100**) |
| 2 клетки | +80 |
| d клеток | `max(foodSignalClose − foodSignalDecay × (d − 1), foodSignalMin)` |
| Минимум | `foodSignalMin` (по умолчанию **+5**), на любом расстоянии |

### Эвристический алгоритм

Базовый алгоритм этапа 7 — **жадный full-board heuristic**:

1. На каждом тике бот оценивает все допустимые абсолютные направления (`up/down/left/right`, кроме разворота на 180°).
2. Для каждого кандидата выполняется проверка немедленной смерти (граница, стена, живая змейка).
3. Для безопасного кандидата считается flood-fill по всей доске (доступная площадь после шага) и штраф тупика.
4. Параллельно оценивается жадная ценность еды:
   - приоритет ближайшей еды;
   - при равной дистанции выше приоритет у еды с большей ценностью очков (`&xN`).
5. Итоговая оценка хода комбинирует:
   - доступную площадь и локальные пути отхода;
   - бонус за еду;
   - штраф за риск входа в узкие области.

Выбирается ход с максимальной итоговой оценкой.

### Тестовая арена ИИ

Модуль арены вынесен в `src/arena/` и поддерживает любые алгоритмы (в т.ч. нейросетевые) через интерфейс `ArenaAlgorithm`.

1. **Headless arena** (без canvas):
   - класс `Arena` — симуляция до смерти, завершения уровня или `maxTicks`;
   - `runOne(maxTicks?)` возвращает фитнес: score, survivedTicks, survivedMs, aliveAtEnd, deathReason;
   - `runArenaSimulation(config)` / `runArenaBatch(config)` — одиночный и пакетный прогоны.
2. **Сравнение алгоритмов**:
   - разные алгоритмы назначаются разным змейкам через `participants`;
   - агрегируются средние показатели по каждому алгоритму.
3. **Demo arena** (с рендером):
   - класс `ArenaDemoRunner`, фабрика `createArenaDemoController(options)`;
   - визуальный запуск на canvas, ускорение `1x / 2x / 4x / 8x`.

| Подсказка | `Пробел — пауза` (или мигающее `⏸ ПАУЗА` если стоит на паузе) |

#### Левая боковая панель

Расположена **слева** от canvas. Отображает статистику **Игрока 1** (первого человека):

```
Имя (bold)
Очки: N
Длина: N
Победы: N
Статус: Жив / [причина смерти]
```

Если игрока 1 нет (0 людей) — панель пуста.

#### Правая боковая панель

Расположена **справа** от canvas. Аналогично, для **Игрока 2**.

#### Нижняя панель (боты)

Расположена **под** canvas. Горизонтальный ряд карточек — по одной на каждого бота. Каждая карточка содержит ту же статистику (имя, очки, длина, победы, статус).

Если ботов нет — панель пуста.

#### Размер canvas

Canvas подстраивается под доступное пространство:
- `cellSize = Math.max(2, Math.min(Math.floor(maxW / boardWidth), Math.floor(maxH / boardHeight)))`
- `maxW` и `maxH` вычисляются из размера окна, за вычетом боковых панелей и (в dev-режиме) ширины dev-панели.

## 12. Режим настройки (dev mode)

### Активация

Dev-режим доступен **только** при сборке с флагом `devmode`:

```bash
npm run dev:debug    # webpack serve --mode development --env devmode --open
```

Webpack определяет глобальную переменную `__DEV_MODE__: boolean` через `DefinePlugin`. В обычной сборке (`npm start`, `npm run build`) переменная равна `false`, и весь код dev-панели не попадает в бандл (tree-shaking).

### Кнопка в меню

В dev-сборке на экране «Меню» появляется дополнительная кнопка **«🛠 Режим настройки»**. При нажатии запускается игра с активным dev-режимом.

### Расположение dev-панели

В dev-режиме игровое поле сдвинуто **влево**, а **справа** от него располагается панель настроек. DOM-структура:

```
.game-layout-row  (flex-строка)
  #game-area      (flex: 1, position: relative — содержит поле, HUD, overlay)
  #dev-panel-container  (фиксированная ширина — содержит панель настроек)
```

**Важно**: dev-панель находится **вне** `#game-area`. Модальные окна (пауза, конец уровня) вставляются внутрь `#game-area` с `position: absolute`. Таким образом, overlay покрывает **только** игровую область, а панель настроек **остаётся кликабельной** во время паузы и модалок.

### Содержимое dev-панели

Панель организована в **секции** (разделы), каждая с заголовком и набором строк «Метка — поле ввода». Все значения привязаны к полям синглтона `gameSettings`.

Для каждого поля (кроме `snakeColors[0..5]`) есть чекбокс **«Для всех уровней»**:
- по умолчанию включён;
- включён → значение считается глобальным (`gameSettings`);
- выключен → значение пишется как per-level override для выбранного уровня (`levelSettingsOverrides[level]`).

Для строк цветов змейки (`snakeColors[0..5]`) чекбокс scope не показывается — эти поля всегда глобальные.

- При каждом «Применить» / «Сохр. ур.» текущие `gameSettings` (включая per-level overrides) сохраняются в `localStorage` под ключом `snake-dev-settings`.
- При открытии dev-панели: если в `localStorage` есть сохранённые настройки → они загружаются и применяются к `gameSettings`.
- «Сброс» очищает `localStorage` и возвращает `gameSettings` к дефолтам из `gameDefaults.json`.

### Экспорт JSON

Экспортируемый файл имеет **ту же структуру**, что и `gameDefaults.json`:

```json
  "scoring": { ... },
  "board": { ... },
  "ai": { ... },
  "colors": { ... },
  "levelOverrides": { "1": { "wallClusters": 5 }, ... }
}
```

Этот файл можно **заменить** вместо `src/engine/gameDefaults.json` в исходниках, чтобы изменить дефолтные параметры игры на постоянной основе.

### Поведение при game over в dev-режиме

При проигрыше в dev-режиме **экран результатов не показывается**. Вместо модалки «Игра окончена» → показывается модалка «Конец уровня», после которой можно продолжить игру (перезапуск того же уровня).

## 13. Конфигурация и параметры

### Единый источник параметров

Все игровые константы хранятся в **одном** JSON-файле:

**`src/engine/gameDefaults.json`**

Этот файл — каноническое хранилище дефолтных значений. Он:
1. Импортируется в `settings.ts` и используется для инициализации `gameSettings`.
2. Может быть заменён файлом, экспортированным из dev-панели.
3. Не содержит кода — только данные.

### Runtime настройки (`gameSettings`)

`gameSettings` — мутабельный синглтон (объект `GameSettings`), инициализируемый из `gameDefaults.json` при загрузке модуля `settings.ts`.

Все игровые системы читают параметры **только** из `gameSettings`, а не из JSON напрямую.

```typescript
export const gameSettings: GameSettings = createDefaultSettings();
```

Dev-панель может изменять поля `gameSettings` в runtime. Функция `resetSettings()` восстанавливает все поля из `gameDefaults.json`.

### Политика ключей конфигурации

- Актуальными считаются только ключи новой схемы (`food*`, `foodCount`, `foodSignal*`, `colors.food*`).
- Legacy-ключи старой схемы (`rabbit*`, `rabbitCount`, `rabbitSignal*`, `colors.rabbit*`) в актуальном ТЗ не используются.
- И frontend-код, и AI/automation-агенты должны опираться только на актуальные ключи новой схемы.

### Структура `gameDefaults.json`

```json
{

    "hungerThreshold": 15,
    "minSnakeLength": 2,
    "initialSnakeLength": 5
  },
```

### Per-level overrides

Объект `levelOverrides` позволяет задать для конкретного уровня фиксированные значения, которые будут использованы **вместо** формул:

```json
"levelOverrides": {
  "3": { "wallClusters": 8, "wallLength": 12, "foodCount": 5 }
}
```

Если для уровня нет override — используется значение из формулы. Overrides поддерживают поля: `wallClusters`, `wallLength`, `foodCount`.

### Все формулы (справочник)

| Формула | Выражение | Параметры по умолчанию | Пример |
|---------|-----------|----------------------|--------|
| Цель уровня | `Math.floor(targetScoreCoeff × level + targetScoreBase)` | 5, 20 | Уровень 1: 25, Уровень 2: 30 |
| Кумулятивная цель | `Σ targetScore(l) для l=1..N` | — | Уровень 3: 25+30+35 = 90 |
| Кластеры стен | `Math.floor(wallClusterCoeff × level + wallClusterBase)` | 1.2, 2 | Уровень 3: 5 |
| Длина стен | `Math.floor(wallLengthCoeff × difficulty + wallLengthBase)` | 1.2, 3 | Сложность 5: 9 |
| Начальная еда | `Math.floor(foodCountPerSnakeCoeff × snakeCount + foodCountBase − difficulty)` | 1.5, 10 | 1 змейка, сложность 1: 10 |
| Размер поля | `base + (level − 1) × increment` | 40, 40, 5 | Уровень 3: 50×50 |

## 14. Игровой цикл (tick-based)

### Общий принцип

Игра работает на **дискретных тиках**. Каждый тик = один шаг симуляции. Между тиками визуальных изменений нет (нет интерполяции).

**Интервал тика**: `tickIntervalMs` (по умолчанию **150 мс** ≈ 6.67 тиков/сек).

Game loop реализован через `setInterval`. При паузе интервал **очищается**, при возобновлении — создаётся заново.

### Таймер уровня

Отдельный `setInterval(1000 мс)` уменьшает `state.levelTimeLeft` на 1 каждую секунду. При паузе этот интервал тоже останавливается.

### Порядок операций в одном тике

Каждый вызов `processTick(state)` выполняет строго в следующем порядке:

```
1. tickCount++

2. Для каждой живой змейки:
   a. Вычислить новую позицию головы
   b. Проверить столкновение со стеной → если да: alive=false, skip
   c. Проверить столкновение с другими змейками → если да: alive=false, skip
   d. Найти еду на позиции новой головы (до перемещения)
   e. Выполнить перемещение (moveSnake, с grow=true если ест)
   f. Проверить самоукус → если да: alive=false, skip
   g. Если ест еду: удалить еду, +scoreValue score, resetHunger

3. Для каждой живой змейки:
   h. processHunger (ticksWithoutFood++, потеря сегмента, возможная смерть)

4. processFoodLifecycle:
   i.  Для всей еды: age++, clockNum++
   j.  Для adult еды: проверка условий, бросок вероятности, спавн потомка
   k.  Добавить новую еду
   l.  Удалить еду с age >= maxAge

5. Пересобрать board[][] из сущностей

6. Проверить checkLevelComplete:
   - Одиночная (классика): score >= cumulativeTarget → levelComplete; !alive ИЛИ успешно завершён уровень 10 → gameOver
   - Одиночная (выживание): score >= cumulativeTarget → мгновенный переход на следующий уровень без паузы; !alive ИЛИ успешно завершён уровень 100 → gameOver (поражение/победа)
   - Мультиплеер: alive <= 1 ИЛИ timeLeft <= 0 ИЛИ успешно завершён уровень 10 → levelComplete/matchEnd
   - Присвоить levelsWon победителю (мультиплеер)
```

### Обработка ввода (вне processTick)

Направления от игроков и ботов применяются **до** вызова `processTick`:

1. `InputHandler` собирает нажатия клавиш и сохраняет последнее валидное направление для каждого игрока.
2. Для ботов вызывается `getBotDirection(state, snake)` → генерируется vision-матрица → `botDecide()` → конвертация в `Direction`.
3. `applyDirection(snake, direction)` — устанавливает направление (с проверкой запрета 180°).
4. Вызывается `processTick(state)`.

### Рендеринг (вне processTick)

После `processTick`:
1. `renderGame(ctx, state, cellSize)` — перерисовка canvas.
2. `renderHUD(...)` — обновление панелей статистики.
3. Проверка `state.levelComplete` → если да, показ модалки.

### Жизненный цикл уровня

```
startGame(config, level)
  → createGameState(config, level)
  → initLevel(state, config)
    → getStartPositions → exclusionZones
    → generateWalls (с зонами безопасности)
    → createSnake × N
    → spawnFood
    → buildBoard
  → buildGameDOM
  → startGameLoop
    → setInterval(processTick + render, tickIntervalMs)
    → setInterval(timer, 1000ms)

// На каждый тик:
gameLoopTick()
  if (paused) return
  applyInputs()
  processTick(state)
  renderGame()
  renderHUD()
  if (state.levelComplete) → handleLevelComplete()

handleLevelComplete()
  → stopGameLoop
  → showLevelCompleteModal / showGameOverModal
  → ожидание Space / кнопки

continueAfterLevelEnd()
  → если game over → finishGame → экран результатов
  → иначе → advanceToNextLevel → startGameLoop
```

## 15 Принципы архитектуры (обязательные)

### 15.1 Цели качества

Код проекта должен быть:

1. **Чистый**: понятные имена, короткие функции, минимум дублирования, единый стиль.
2. **Поддерживаемый**: изменения в одной подсистеме не должны требовать правок во многих файлах.
3. **Тестируемый**: ключевая логика engine покрывается unit-тестами без DOM/Canvas; тесты детерминированы.
4. **Расширяемый**: новые механики/сущности/экраны добавляются через новые модули/системы, а не через разрастание условий в центральных файлах.
5. **Читаемый**: управление жизненным циклом игры и UI (экраны/модалки/пауза) описано явно и однозначно, без противоречивых комбинаций флагов.
6. **Слабо связанный**: доменная логика (engine) не импортирует и не использует адаптеры окружения (DOM, Canvas, localStorage, таймеры).

---

### 15.2 Разделение на слои (Layering)

#### Engine (домен)

1. Engine — чистая логика симуляции: состояние, формулы, движение, коллизии, спавн, механики.
2. Engine **не использует**: DOM API, Canvas API, `localStorage`, `setInterval`, прямой `Math.random()`.
3. Engine допускает только:
   - чтение/мутацию `GameState`;
   - генерацию доменных событий (см. 15.6);
   - чтение параметров только из `gameSettings`/`EngineContext`.

#### App/UI (приложение)

1. App/UI отвечает за:
   - роутинг по экранам;
   - обработку пользовательского ввода;
   - показ модалок и управление паузой;
   - запуск/остановку game loop;
   - работу со storage (рекорды, имена, dev-настройки).
2. App/UI **не мутирует** `GameState` напрямую (см. 15.4).

#### Renderer

1. Renderer читает состояние и рисует; **не содержит логики** симуляции и не мутирует `GameState`.
2. Любые визуальные эффекты (если появятся) хранятся в отдельном “visual state” renderer-слоя и **не влияют** на правила игры.

---

### 15.3 Порты и адаптеры (Hexagonal / Ports & Adapters)

1. Все зависимости на окружение оформляются как **порты (интерфейсы)** в Engine и реализуются **адаптерами** в App/Infra.
2. Минимально обязательный порт: `RandomPort` (RNG).
   - Все операции случайности в Engine используют `ctx.rng.next()`.
   - Прямой `Math.random()` в модулях `engine/**` запрещён.
3. Дополнительные порты (рекомендуются): `StoragePort`, `SchedulerPort`/`ClockPort`, `LoggerPort`.

---

### 15.4 Единый источник истины (Single Source of Truth) — вариант A

1. `GameState` — **единственный источник истины** для текущей партии (позиции, очки, возраст, статусы и т.д.).
2. Мутация `GameState` разрешена только:
   - внутри функций Engine (`initLevel`, `processTick`, системы);
   - через единый use-case слой приложения (например `GameController`), который вызывает Engine в правильном порядке.
3. UI/Renderer/InputHandler **не имеют права** мутировать `GameState` напрямую.
4. Производные структуры (`board[][]`, occupancy-кэши и т.п.) должны быть **пересчитываемыми** и не являются мастер-данными.

---

### 15.5 Pipeline / Systems (конвейер систем тика)

1. `processTick` реализуется как последовательность **систем** (pipeline), каждая отвечает за одну механику/фазу.
2. Система имеет единый контракт (рекомендуется): `run(state, ctx, out)` и не взаимодействует с DOM/Canvas.
3. Порядок выполнения систем соответствует регламенту тика из раздела 14 и задаётся явным списком систем.
4. Добавление новых механик выполняется через:
   - добавление новой системы;
   - или расширение одной системы;
   без разрастания монолитной функции.

---

### 15.6 Доменные события (Domain Events)

1. Engine не вызывает UI/звуки/анимации напрямую. Вместо этого Engine формирует **Domain Events** (события фактов).
2. `processTick` возвращает `TickResult { events: DomainEvent[] }` (или эквивалентный механизм), содержащий события тика.
3. Примеры доменных событий (минимальный обязательный набор):
   - `SNAKE_DIED` (snakeId, reason)
   - `RABBIT_EATEN` (snakeId, pos, newScore)
   - `RABBIT_BORN` (parentPos, childPos)
   - `LEVEL_COMPLETED` (reason, winnerId?)
   - `GAME_OVER` (если применимо)
4. App-слой обрабатывает Domain Events и выполняет побочные эффекты: модалки, звук, storage, переходы экранов.

---

### 15.7 Ввод пользователя: Command Buffer + применение на границе тика

1. Обработчики DOM-событий не меняют `GameState`. Они складывают команды в буфер (`InputBuffer`).
2. На каждом тике команды **потребляются** и применяются строго в начале тика (в соответствии с правилами ввода из раздела 3 и регламентом тика из раздела 14).
3. Буфер ввода должен поддерживать правило: “последнее валидное направление за тик” и запрет разворота на 180°.

---

### 15.8 UI/экраны/пауза: конечный автомат (FSM)

1. Логика экранов и модалок реализуется через **Finite State Machine (FSM)**.
2. `Space` и кнопки работают как события FSM; поведение Space определяется текущим состоянием автомата (раздел 3. Пауза).
3. Противоречивые комбинации флагов (например, “paused=true и одновременно модалка конца уровня”) не допускаются; валидность обеспечивается моделью состояний.
4. Переходы (menu → game → results, playing ↔ paused, levelComplete, gameOver) описываются явно (таблица переходов или reducer-функция).

---

### 15.9 Контекст зависимостей (явная DI без контейнеров)

1. Engine получает зависимости через `EngineContext` (например: `settings`, `rng`, опционально `scratch`).
2. Тесты могут подменять зависимости (seeded RNG, настройки) без моков DOM.

---

### 15.10 Derived-данные и оптимизации

1. `board[][]` и любые occupancy-индексы являются derived-структурами и могут пересобираться из мастер-данных (по ТЗ — каждый тик).
2. Коллизии и правила симуляции должны опираться на мастер-данные (`snakes`, `foodItems`, `walls`) либо на корректно синхронизированные derived-индексы, построенные из мастер-данных в пределах тика.

---

### 15.11 Обязательные OOP-требования

1. Ключевые доменные сущности реализуются как классы с инкапсулированным поведением (например, `SnakeEntity`, `FoodEntity`), а не как набор разрозненных процедур.
2. Файл `engine/types.ts` содержит только типы/интерфейсы TypeScript. Конкретные реализации классов в него не помещаются.
3. Публичный API классов должен выражать доменные действия (`move`, `applyDirection`, `die`, `tickLifecycle`), а не технические детали хранения.
4. Инварианты сущностей (запрет разворота на 180°, корректная смерть, корректное изменение длины, счётчиков и т.п.) поддерживаются внутри методов классов.
5. Оркестрация use-case уровня (`GameController`, `SnakeGameApplication`) не содержит доменных вычислений сущностей, а координирует вызовы сервисов/engine.
6. Классы должны соответствовать SRP: при росте сложности ответственность выносится в отдельные сервисы/координаторы, а не накапливается в одном «god object».
7. Предпочтение отдаётся композиции над наследованием; наследование допускается только при явной и устойчивой модели «is-a».
8. Зависимости передаются явно (DI через контекст/конструктор); скрытые глобальные зависимости внутри классов не допускаются.


## 16. Тесты (Jest)

### Общие требования

- Среда: **jsdom** (для тестов, использующих DOM API).
- CSS-импорты мокаются через `styleMock.js`.
- Все engine-тесты **не зависят** от DOM — тестируют чистую логику.
- Формат: `describe` + `it`/`test`, с понятными описаниями на английском или русском.
- Перед каждым тестом: `resetSettings()` для чистого состояния.

### Обязательные группы тестов (Этап 3c)

#### 16.1 Движение змейки

| Тест | Что проверяется |
|------|-----------------|
| `getNextHeadPosition` вычисляет правильную клетку для каждого направления | `up` → y−1, `down` → y+1, `left` → x−1, `right` → x+1 |
| `moveSnake(grow=false)` добавляет голову и удаляет хвост | Длина не меняется, голова на новой позиции |
| `moveSnake(grow=true)` добавляет голову, хвост остаётся | Длина += 1 |

#### 16.2 Запрет разворота на 180°

| Тест | Что проверяется |
|------|-----------------|
| `isReverseDirection('right', 'left')` → `true` | Противоположные направления определяются |
| `isReverseDirection('right', 'up')` → `false` | Поворот на 90° разрешён |
| `applyDirection` с разворотом не меняет direction | Змейка сохраняет прежнее направление |
| `applyDirection` с поворотом на 90° меняет direction | Направление обновляется |

#### 16.3 Коллизии

| Тест | Что проверяется |
|------|-----------------|
| `collidesWithWall` при выходе за границы → `true` | Позиция `(-1, 5)` или `(width, 0)` |

| `collidesWithWall` на свободной клетке → `false` | |
| `selfCollision` при наличии совпадения головы с сегментом → `true` | |
| `selfCollision` при отсутствии совпадения → `false` | |
| `collidesWithSnake` с другой змейкой → `true` | Позиция совпадает с сегментом другой живой змейки |
| `collidesWithSnake` с мёртвой змейкой → `false` | Мёртвые змейки не учитываются |

#### 16.4 Голод

| Тест | Что проверяется |
|------|-----------------|
| `processHunger` после `hungerThreshold − 1` тиков без еды — змейка цела | `ticksWithoutFood` инкрементируется |
| `processHunger` на тике `hungerThreshold` — теряет 1 сегмент | `segments.length` уменьшается на 1, `ticksWithoutFood = 0` |
| Повторная потеря сегментов до `minSnakeLength` → `alive = false` | `deathReason = 'Умерла с голоду'` |
| `resetHunger` сбрасывает `ticksWithoutFood` в 0 | |

#### 16.5 Поедание еды

| Тест | Что проверяется |
|------|-----------------|
| Змейка съедает еду → `score += food.scoreValue` | |
| Змейка растёт на `food.growthValue` сегментов | `segments.length` увеличивается |
| Еда удаляется из `state.foodItems` | |
| `ticksWithoutFood` сбрасывается | |

#### 16.6 `processTick` (интеграционный)

| Тест | Что проверяется |
|------|-----------------|
| Один полный тик: змейка двигается на 1 клетку | Новая позиция головы, хвост удалён |
| Тик со столкновением со стеной → `alive = false` | `deathReason = 'Врезалась в стену'` |
| Тик с поеданием еды → рост + очки | |
| `tickCount` увеличивается | |

#### 16.7 Спавн еды

| Тест | Что проверяется |
|------|-----------------|
| `spawnFood` возвращает запрошенное количество еды | Или меньше, если не хватает места |
| Ни одна еда не на стене | `food.pos` не совпадает с `state.walls` |
| Ни одна еда не на змейке | `food.pos` не совпадает с сегментами |
| Расстояние Чебышёва между любыми сущностями еды > 1 | |
| Вся еда внутри границ поля | |
| `age`, `clockNum`, `reproductionCount` инициализированы корректно | |

#### 16.8 Генерация стен (BFS-связность)

| Тест | Что проверяется |
|------|-----------------|
| `validateWalls` для пустого списка стен → `true` | |
| `validateWalls` для стен, создающих изолированную зону → `false` | Стена от края до края |

#### 16.9 Формулы

| Тест | Что проверяется |
|------|-----------------|

| `getCumulativeTargetScore(3)` = 25 + 30 + 35 = 90 | |
| `getWallClusterCount(3)` = `Math.floor(1.2 × 3 + 2)` = 5 | |

| `getInitialFoodCount(1, 1)` = `Math.floor(1.5 × 1 + 10 − 1)` = 10 | |
| Все формулы используют `Math.floor` | |

| Тест | Что проверяется |
|------|-----------------|
| Одиночная: `score >= cumulativeTarget` → `true` | |

## 18. Актуальные уточнения и журнал решений

Раздел фиксирует только актуальные уточнения, которые уже интегрированы в основное ТЗ.

1. Максимум людей-игроков: **2**. Режим наблюдателя (0 людей) поддержан.
2. Округление формул: всегда `Math.floor()`.
3. Лимит уровней:
   - классика (1 игрок) — до уровня **10**;
   - мультиплеер/смешанный режим — до уровня **10**;
   - выживание (1 игрок) — до уровня **100**, с автоматическим переходом по достижению цели уровня без паузы.
4. Еда:
   - единая модель `Food` + наследники (`AppleFood`, `RabbitFood`);
   - возраст и фазы — в базовом классе;
   - размножение — общее поведение базового класса с возможностью переопределения в наследнике;
   - логика, ранее описанная для кроликов, трактуется как общее правило для еды;
   - при поедании любой еды голод змейки сбрасывается.
5. Яблоки (базовая еда):
   - `young` (зелёное): +1 очко, +1 сегмент;
   - `adult` (красное): +2 очка, +2 сегмента;
   - `old` (коричневое): +1 очко, +1 сегмент.
6. Автодобавление взрослой еды:
   - при `foodCount < aliveSnakeCount`;
   - позиция выбирается максимально удалённой от голов змеек по Чебышёву;
   - при равенстве предпочтение равноудалённым позициям;
   - не чаще 1 раза за `hungerThreshold` тиков.
7. HUD:
   - структура и размещение остаются как в базовой версии (P1 слева, P2 справа, боты снизу);
   - ориентация HUD остаётся в мировых координатах (без поворота под стороны игроков).
8. Ключи конфигурации:
   - в актуальной версии ТЗ используются только ключи новой food-схемы;
   - legacy rabbit-ключи считаются устаревшими и не применяются в новой разработке.
